Alaya · Architecture Manifesto

Operational memory для любого ИИ-агента.

«Из разговоров — в состояние.
Из состояния — в действия.»

Alaya непрерывно поглощает рабочие данные компании и превращает их в структурированное состояние мира с провенансом и темпоральностью. Поверх этого состояния — движок workflows: reminders, follow-ups, stale-detection. Любой агент через CLI, MCP или REST работает не с сырым шумом, а с уже собранной моделью и запланированными действиями.

01 · Проблема

ИИ-агент знает, что сказали. Не знает, что осталось сделать.

Решения теряются в истории чатов. Обещания не отслеживаются. Устаревшие факты маскируются под актуальные. Каждый агент собирает контекст руками и начинает каждую сессию с нуля. Попытки навести порядок через wiki проваливаются — стоимость поддержания растёт быстрее её ценности.

Memory-only системы

RAG достаёт чанки и забывает. Vector store ищет похожее. Mem0/Letta хранят факты. Но никто не отвечает на «что я обещал и кому, срок — завтра?».

Workflow-only системы

Lindy, Zapier, Granola запускают действия. Но они не ведут модель мира — не отличают факт от гипотезы, не видят устаревания, не связывают новое с предыдущим.

Разрыв между ними

Память без execution — аналитика. Execution без памяти — слепой cron. Alaya стоит ровно на этом шве: изменение состояния → триггер действия.

Корневая мысль Реальная ценность для компании — не «помню больше текста», а «ничего важного не падает между стульями». Это требует трёх свойств одновременно: памяти с провенансом, темпоральной модели мира, и execution-слоя, запускающегося от изменений состояния.
02 · Рамка

Не память. Не workflows. Operational memory + workflow engine.

Alaya мыслится не как база знаний и не как RPA-платформа. Это единый слой, в котором поток событий превращается в рабочее состояние мира, а изменения состояния запускают действия. Четыре слоя, один контракт.

1 · Events

Сырые сообщения, транскрипты, тикеты, обновления из интеграций. Неизменяемые, с метаданными источника и ACL.

2 · Memory

Сущности, факты, связи, обязательства, предпочтения, цели. Declared vs inferred. Темпоральные, с провенансом.

3 · Reasoning

Извлечение, матчинг, нормализация, консолидация, разрешение противоречий. LLM под контролем схемы.

4 · Action Loop

Reminders, delayed jobs, follow-ups, stale-detection, review-циклы. Workflows, триггерящиеся от изменения состояния.

Почему это важно Memory-only вендоры решают часть (1)+(2). Workflow-вендоры — часть (4). Но реальные рабочие сценарии — обязательства, статусы, незакрытые темы — живут ровно на стыке. Наша ставка: если соединить четыре слоя в один контракт, получится продукт, которого пока нет.

Два принципа, без которых рамка не работает

Declared vs Inferred

Что человек явно сказал — это одно. Что модель вывела из контекста — это другое. В Alaya эти два класса фактов различаются на уровне схемы: у каждого claim'а есть kind. Inferred не может маскироваться под declared. Correction loop уважает это различие.

Provenance как обязательное свойство

Каждый derived факт связан с исходными событиями через extraction_run_id. Без provenance память — чёрный ящик. С provenance она проверяема, объяснима и пригодна для correction. Цитата в ответе агента — это не украшение, а контракт.

03 · Слои памяти

От сырого события до оглавления как книги.

Каждый факт в Alaya проходит путь от реплики в чате до узла в knowledge tree. Разделение на слои — не метафора, а физический контракт: у каждого слоя своя таблица, свой SLA на свежесть, свой способ запроса.

04 · Pipeline извлечения

От сырого события до графа — за четыре шага.

Pipeline разнесён на четыре стадии, чтобы каждая использовала модель подходящей силы. Дешёвая отбрасывает шум. Дорогая работает только по содержательному. Результат дешевле одной-сквозной экстракции минимум на 40%, при сопоставимом качестве.

Haiku · дёшево

Cortex

Режет событие на семантические чанки и классифицирует каждый: narrative, list, table, smalltalk. Двухпроходная верификация отбрасывает шум.

→ L0 Chunks + domain scores
Sonnet · точно

Crystallizer

На каждом «crystal»-чанке извлекает entities, claims, relations с confidence-тирами и kind. Второй проход сравнивает, фиксирует расхождения.

→ ExtractionResult verified
Resolver · атомарно

Writer

Разрешает сущности против графа: exact → fuzzy → LLM. Атомарно пишет под workspace-lock с полным провенансом. Пропагирует ACL от event к claim.

→ Entities · Claims · Relations
Sonnet · фоново

Integrator

Дедупликация (3-tier), обогащение связей, нормализация дат, contradiction-detection. Работает по dirty-set с max-wait 30 мин.

→ Интегрированный граф

Четыре контура защиты от «яда» в памяти

Sanitizer на входе

NFKC-нормализация, стрипинг zero-width символов, удаление HTML-комментариев, детектирование injection-паттернов. Подозрительное логируется в monitoring.

XML-границы prompt'а

Систем-промпт использует <instructions>, <ontology>, <data> — инструкции физически отделены от данных, LLM знает что где.

Pydantic-валидаторы

Весь output LLM проходит через схему: script-теги отклоняются, control-символы стрипаются, confidence зажимается в [0, 1], длина ограничена.

Job-контракт

TaskIQ-задачи принимают только string-ID. Сырой текст никогда не ходит по очереди — воркер читает его из защищённой БД.

Базовый принцип Output LLM — это ненадёжный пользовательский ввод. Его нельзя доверять в обход схемы. Любой факт в графе — это то, что прошло через валидатор, а не то, что сказала модель.
05 · Action Loop

Память, которая напоминает сама.

Memory без execution — полезный, но неполный слой. Alaya встраивает workflow-движок рядом с графом: изменения состояния триггерят действия. Это то, что отличает operational memory от обычной — и то, чего нет ни у одного из «памятевых» вендоров.

Reminders

Триггер: новый commitment-claim с due_date.

Действие: запланировать job_remind на due - 1d. Агент/человек получает напоминание через MCP / Slack / email.

Idempotency: по (claim_id, kind=remind).

Follow-ups

Триггер: незакрытая тема — inferred claim со статусом unresolved, старше N дней.

Действие: создать review-item с провенансом исходного обсуждения.

Отмена: если появился новый claim, closing эту тему.

Stale detection

Триггер: max(valid_from, last_touched) + decay(trust) > threshold.

Действие: пометить claim как stale, поднять в weekly review, снизить вес в retrieval.

Возврат: свежий upsert сбрасывает флаг.

# 2026-04-01: извлечён commitment claim(kind=declared, commitment: Ivan → release beta by 2026-04-15) valid_from: 2026-04-01 source: meeting/notes/17 # Action Loop триггер → enqueue job_remind at (due - 1d) = 2026-04-14 09:00 → enqueue job_followup_check at (due + 1d) = 2026-04-16 09:00 # 2026-04-14 09:00: reminder fires → MCP notify: "Beta release — tomorrow. Status?" # 2026-04-16: follow-up check → query claims about Beta release since 2026-04-14 → если нет closing-claim: поднять в review
Архитектурное решение TaskIQ уже в стеке как исполнитель extraction-pipeline. Action Loop переиспользует ту же инфраструктуру: delayed jobs, recurring, retries, idempotency — бесплатно. Новое здесь — не машина исполнения, а контракт триггеров от состояния памяти.
Граница между core и paid tier В open-core остаются примитивы: триггеры, reminder-jobs, stale-detection, follow-up-механика. Готовые бизнес-процессы и domain-recipe'ы (sales stage transitions, compliance review cycles, onboarding progression) — это отдельный SaaS Alaya Workflow, который работает stateless через API-ключ к клиентскому Core. 06 · Контракт с агентом

Агент — клиент графа, а не владелец памяти.

В корпорации граф один — это общая правда компании. Но разным агентам нужна разная проекция. Это решается не моделью «агент как личность», а тремя простыми механиками поверх единого графа.

Scoped API keys

У каждого ключа — набор scope'ов и фильтров: entity_types, tags, read|write|admin. Один граф, но агент видит только свою линзу. RLS снизу гарантирует workspace-изоляцию; scopes сверху — ролевую.

Person briefing

При запуске агента от имени конкретного человека — Alaya отдаёт person-centric проекцию: чем человек занят, к каким проектам привязан, какие commitments на нём висят. Dev получает свой срез, PM — свой, CEO — свой.

MCP wake-up

На MCP-коннекте агент получает ~170 токенов сжатого wake-up контекста: имя воркспейса, ключевые сущности, свежие изменения, открытые commitments. Достаточно, чтобы не грузить сессию с нуля.

07 · Progressive disclosure

Агент получает ровно столько контекста, сколько нужно.

Наивный подход — грузить всё и фильтровать в LLM. Это дорого, медленно и ломается на большом графе. Alaya отдаёт контекст слоями — от сжатого приветствия до полного deep-search по L0. Агент сам решает, на какой уровень спуститься.

Слой 1Wake-up
Имя воркспейса, счётчики ключевых сущностей, последние 3 изменения, свои открытые commitments. Достаточно чтобы агент «понимал где он».
~170токенов
Слой 2Summary
Топ-L1 сущностей по релевантности + свежие L2 claims. Отвечает на «кто-что-когда» без похода в сырые данные.
500–800токенов
Слой 3Scoped query
Поиск по конкретной сущности/предикату. Возвращает claims + relations + стейт Action Loop с evidence. Закрывает ≥80% реальных вопросов агента.
2–4Kтокенов
Слой 4Deep search
Полный 3-канальный RRF по всему графу, включая L0-события. Для сложных кейсов: «что именно было сказано в переписке на прошлой неделе?».
до лимитаокна
Экономика 90% запросов агента укладывается в первые два слоя — это ~1K токенов против ~50K на «сырой RAG со всем на свете». На миллионе запросов в месяц разница — в счёте на LLM порядки, не проценты.
08 · Безопасность

Централизованная память — одна точка входа.

Общий граф знаний компании — привлекательная цель. Tenant-изоляция, injection-защита, memory poisoning, attribution — угрозы распределены по слоям, и защита тоже.

Tenant isolation — защищено схемой

PostgreSQL Row-Level Security

14 таблиц с FORCE ROW LEVEL SECURITY. На каждом запросе SET LOCAL app.workspace_id. Даже если приложение забудет фильтр — БД вернёт пустой результат.

Composite FK (workspace_id, id)

Все дочерние таблицы ссылаются на родителя парой ключей. Cross-workspace reference физически невозможен на уровне схемы.

09 · Продуктовая линия

Память — open-core. Автоматизация и sync — paid tiers.

Alaya разбивается на три продукта над одним контрактом. Основа — open-core память с workflow-примитивами. Поверх неё — два коммерческих слоя: stateless-оркестровка бизнес-процессов и локальный sync-демон для AI-инструментов пользователя. Каждый следующий слой работает через API-ключ к клиентскому Core и не дублирует storage.

Open-core · BSL 1.1

Alaya Core

Серверная память и примитивы исполнения. Self-host бесплатно, managed cloud — по подписке.

  • Event ingestion + connectors
  • Multi-stage extraction
  • Hybrid retrieval + tree
  • TaskIQ workflow primitives
  • MCP / REST API контракт
SaaS · paid · stateless

Alaya Workflow

Оркестровка бизнес-процессов поверх Core. Не хранит клиентских данных — работает через API-ключ к Core клиента.

  • BYO Core (self-host или managed)
  • Ничего не persists на стороне Workflow
  • Domain packs: sales / ops / HR / compliance
  • Visual triggers + recipe-библиотека
  • Webhooks и мост к внешним системам
Desktop · paid tier

Alaya Sync

Локальный демон для синхронизации AI-инструментов с корпоративной памятью. Делает sync невидимым для юзера.

  • Claude Code / Cursor / Zed AI / Raycast
  • Briefing inject на SessionStart
  • Capture PreCompact / SessionEnd
  • Верификация цитат и writebacks
  • Auto-update под новые AI-tools

Зачем такое разделение Open-core память остаётся доступной — это распространение и доверие. Workflow и Sync — два разных класса commercial-продуктов: первый решает «как автоматизировать бизнес-процесс поверх памяти», второй — «как сделать так, чтобы память всегда была актуальной без дисциплины со стороны юзера». Ни один не дублирует storage — оба работают через API клиентского Core.
10 · Карта рынка

Где сильны другие. Чем отличается Alaya.

Мы не изобретаем память с нуля — рынок зрелый. Но в выбранной комбинации свойств пересечение конкурентов минимально. Ниже — где конкретно проходят границы.

Memory middleware
Layer под агентом
Mem0 — memory middleware, key-value поверх vector store.
Letta / MemGPT — stateful agent runtime с memory-блоками.
Supermemory — API для персональной памяти агента.
Сильны в хранении фактов. Но нет темпоральной модели, нет declared/inferred различения, нет execution-слоя. Alaya добавляет workflow engine поверх памяти.
Graph-native memory
Temporal knowledge graphs
Graphiti / Zep — bi-temporal knowledge graph, invalidation вместо deletion.
Cognee — knowledge pipeline с graph-представлением.
Сильны в темпоральной истине и графовых связях. Но graph-first архитектура — тяжёлая база без доказанной пользы на всех use-case'ах. Alaya — Postgres-first с опциональным graph-слоем, плюс Action Loop.
Enterprise search
«AI-поиск по корпорации»
Glean — enterprise search + AI assistant.
Dust — workspace assistants с retrieval.
Сильны в retrieval по корпоративным источникам. Но это чистый RAG — нет модели мира, нет commitments, нет reminders. Alaya ведёт состояние, а не только отвечает на вопросы.
Workflow / agent shells
Движки автоматизации
Lindy — workflow automation с LLM.
Granola — meeting assistant с action-items.
Сильны в execution. Но у них нет памяти как first-class — workflow живёт в моменте, не в state. Alaya триггерит действия от изменений в графе, а не от внешних событий.

Почему комбинация работает Повторить отдельный слой может любой. Повторить связную композицию — память + provenance + declared/inferred + темпоральность + state-triggered workflows + agent-agnostic API — быстро и качественно трудно.
11 · Roadmap

Что уже работает и что в очереди.

Alaya развивается по run'ам. Каждый run — одна логическая способность, PR-per-sprint, dual-model review перед мержем.

Run 1 · Foundation

Shipped

Монорепо, 18 таблиц, RLS на 14 из них, REST API, API-keys c хэшем, error-envelope, курсор-пагинация, CI.

Run 2 · Extraction

Shipped

Single-pass Sonnet extraction, TaskIQ workers, Anthropic adapter с ephemeral cache_control, FakeLLMAdapter.

Run 3 · Intelligence pipeline

Shipped

Cortex (Haiku) → Crystallizer (Sonnet) → Integrator с двухпроходной верификацией, 3-tier resolver, dedup, date-normalizer. ≥40% экономии на LLM.

Run 4 · Go CLI

Shipped

Single-binary Cobra CLI. OS keyring, dual-mode output, setup agent --profile claude-code.

Run 5 · Search & Ask & Tree

Shipped

Hybrid 3-channel RRF, Ask с валидацией цитат, Knowledge tree с dirty-flag rebuild. Embedding pipeline на multilingual-e5-large.

Run 5.1 + 5.2 · Hardening

Shipped

DB-backed workspace lock, scope enforcement, fail-closed rate-limit, docs off в prod, TrustedHost middleware.

Run 6 · MCP Server

Next

MCP wake-up (~170 токенов), Person briefing, API-key scope filters, related-paths в tree, content-policy validator.

Run 7 · Action Loop MVP

commitment entity type, declared vs inferred в схеме claims, job_remind/job_stale_scan/job_followup, ACL propagation, correction loop. Vision alignment P1.

Run 8 · Connectors

Slack, GitHub, Linear, Webhook. Delta-sync с ETag, ACL-mapping источника в entity access_level.

Run 9 · Web UI

Graph browser, live extraction runs monitor, workspace admin, review-queue для stale/follow-ups.

Contradiction detection, forgetting-decay по trust, trust-feedback, ретроспективный memory-poisoning scan, LongMemEval harness.

12 · Что здесь нашего

Know-how не в одной фиче. В композиции.
