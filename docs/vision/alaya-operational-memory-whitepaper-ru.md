# Alaya как Operational Memory and Workflow Engine for AI Assistants

Черновик white paper  
Язык: русский  
Статус: exploratory draft

## Кратко

Идея Alaya в том, чтобы быть не просто "памятью для агента" и не просто "чат-ботом", а слоем, который:

- принимает события из разговоров, чатов, встреч и интеграций
- превращает их в структурированное состояние
- понимает, что изменилось в мире пользователя или команды
- запускает нужные напоминания, действия и workflows

Если сформулировать совсем просто:

**Alaya превращает разговоры в состояние, а состояние в действия.**

Это и есть смысл формулы:

**Operational Memory and Workflow Engine for AI Assistants**

По-русски:

**Операционная память и движок workflows для AI-ассистентов.**

## Почему обычной "памяти" мало

Сегодня многие AI-системы говорят, что у них есть память. Чаще всего под этим имеется в виду одно из трех:

- история сообщений
- векторный поиск по старым данным
- набор сохраненных фактов о пользователе

Этого достаточно, если задача только в том, чтобы "вспомнить что-то похожее".

Но этого недостаточно, если ассистент должен:

- отслеживать договоренности
- понимать, что обещание еще не выполнено
- замечать, что информация устарела
- различать "человек сказал" и "модель предположила"
- вовремя напоминать
- запускать follow-up действия
- связывать новые события с уже известными целями, людьми и темами

То есть нам нужна не только память как хранилище, но и память как рабочая система.

## Что такое operational memory

Operational memory это память, которая хранит не просто текст, а рабочее состояние мира.

Она отвечает не только на вопрос:

- "Что я знаю?"

Но и на вопросы:

- "Что сейчас важно?"
- "Что было обещано?"
- "Что не закрыто?"
- "Что протухает без внимания?"
- "Что нужно сделать дальше?"
- "Какое действие должно быть запущено автоматически?"

## Основная гипотеза

Следующая волна AI-ассистентов будет строиться не вокруг одного большого чата, а вокруг четырех слоев:

1. `Events`  
Сырые события: сообщения, транскрипты, аудио, реакции, документы, встречи, обновления из интеграций.

2. `Memory`  
Структурированная память: сущности, факты, связи, договоренности, предпочтения, цели, даты, незакрытые темы.

3. `Reasoning`  
LLM и логика, которые интерпретируют изменения, извлекают смысл, разрешают неоднозначность и помогают принять решение.

4. `Action Loop`  
Напоминания, delayed jobs, recurring jobs, workflows, follow-ups, recovery-механизмы, чтобы ничего важного не терялось.

У большинства систем сегодня хорошо развиты только отдельные части этой цепочки.  
Alaya потенциально интересна тем, что соединяет их вместе.

## Базовая архитектура Alaya

В обсуждении выделилась архитектурная линия, которая выглядит сильной и переносимой между доменами.

### 1. L0 Event Log

Все входящие данные сначала сохраняются как неизменяемые сырые события.

Примеры:

- сообщение в Telegram
- сообщение в Slack
- транскрипт встречи
- summary из PLAUD
- голосовое сообщение после локальной транскрибации
- изменение задачи или календарного события

Смысл L0:

- ничего не теряется
- всегда можно показать исходный источник
- удобно переобрабатывать данные при улучшении extraction pipeline

### 2. Chunking и первичная классификация

Сырые события делятся на смысловые куски.  
После этого система решает, какие куски достаточно важны, чтобы отправлять их дальше в extraction.

Это важно по двум причинам:

- экономия на моделях
- снижение шума

### 3. Extraction

На этом этапе из текста извлекаются:

- сущности
- факты
- связи
- статусные изменения
- договоренности
- риски
- решения
- action items

Ключевой момент: extraction должен быть не "магией", а управляемым слоем со схемами, confidence и provenance.

### 4. Canonical State

Извлеченные данные не должны жить только в виде разрозненных кусочков памяти.

Они должны собираться в каноническое состояние:

- кто есть кто
- какие есть цели
- какие есть отношения между объектами
- что уже подтверждено
- что только inferred
- что устарело
- что требует follow-up

### 5. Workflow Engine

Когда состояние меняется, это может запускать действия.

Например:

- найдено обещание -> поставить follow-up
- приближается важная дата -> отправить reminder
- найден конфликт или незакрытая тема -> добавить в weekly review
- сущность давно не обновлялась -> пометить как stale
- пользователь исправил факт -> пересчитать downstream state

Это превращает memory system в operational system.

## Главная особенность: declared vs inferred

Одна из самых сильных идей в Alaya-подходе:

- `declared state` — то, что человек явно сказал
- `inferred state` — то, что система вывела из контекста

Пример:

- declared: "Я почти закончил"
- inferred: задача все еще заблокирована

Или:

- declared: "Все нормально"
- inferred: тема остается напряженной и не закрытой

Сохранять оба слоя важно, потому что:

- это честнее
- это лучше для отладки
- это позволяет не путать факт и гипотезу
- это снижает риск ложной уверенности системы

## Provenance как обязательное свойство

Provenance означает, что каждый важный вывод можно связать с исходными событиями.

Иначе говоря, система должна уметь ответить:

- откуда мы это взяли
- из какого сообщения или разговора это появилось
- когда это было выведено
- насколько это надежно

Без provenance память легко превращается в непрозрачный black box.

С provenance память становится:

- проверяемой
- объяснимой
- пригодной для correction loop

## Почему workflows здесь так важны

Во многих системах память заканчивается на retrieval:

- найди похожие события
- покажи похожие факты
- подставь это в prompt

Но в реальном продукте ценность часто возникает дальше:

- напомнить вовремя
- не забыть договоренность
- отследить обязательство
- сделать follow-up
- не дать важной теме исчезнуть

Поэтому delayed tasks, recurring schedules, retries, recovery cron, idempotent execution и memory-triggered workflows здесь не второстепенны.  
Они являются частью самой ценности.

## Чем Alaya может отличаться от других memory systems

На рынке уже есть сильные игроки:

- memory middleware
- stateful agent platforms
- graph memory systems
- vector + graph pipelines

Но Alaya может занять другую позицию.

Не:

- "еще одна память"
- "еще один graph"
- "еще один агентный фреймворк"

А:

**операционная память для ассистентов, которая соединяет память и выполнение действий**

То есть Alaya полезна не только для recall, но и для:

- commitments
- follow-through
- reminders
- status transitions
- stale state detection
- operational reviews
- domain workflows

## Под какие продукты это подходит

Такой core может лежать под разными vertical products:

- корпоративный chief of staff
- relationship copilot
- personal life assistant
- coaching assistant
- recruiting memory assistant
- account manager copilot
- knowledge + workflow assistant для команд

Общая черта у этих продуктов одна:

**они живут не только на поиске по памяти, но и на сопровождении обязательств, состояний и следующих действий.**

## Что в этой идее может быть know-how

Know-how здесь не в одной секретной функции, а в сочетании нескольких слоев:

- event-sourced memory
- typed extraction
- canonical entities
- declared vs inferred state
- provenance
- temporal updates
- workflow triggering от состояния памяти
- delayed and recurring execution

Эту идею можно повторить.  
Но сложно быстро и хорошо повторить всю систему целиком, особенно если она реально работает на messy conversational data.

## Куда это можно развивать

Если смотреть вперед, Alaya имеет смысл развивать в сторону:

### 1. Memory-aware workflows

Не просто cron по времени, а workflows, которые зависят от состояния памяти.

Примеры:

- если обещание не закрыто 5 дней -> отправить follow-up
- если confidence низкий -> спросить уточнение вместо действия
- если факт устарел -> пометить и вынести в review
- если найдено противоречие -> не действовать автоматически

### 2. Domain packs

Один и тот же core можно настроить под разные домены:

- company
- relationship
- coaching
- personal
- recruiting

Это делает систему универсальнее без потери глубины.

### 3. Correction loop

Пользователь должен иметь возможность исправлять память, а система должна уметь:

- обновлять состояние
- помечать старые выводы как superseded
- корректировать downstream workflows

### 4. Evaluation layer

Для operational memory критично измерять:

- точность extraction
- точность commitments/follow-ups
- полезность reminders
- долю stale или contradicted state
- качество provenance

### 5. Agent-agnostic interfaces

Core должен уметь работать не только как часть одного приложения, но и как внешний слой для других систем:

- API
- SDK
- MCP server
- adapters для внешних agent runtimes

## Практическая формула

Если попробовать свести идею к короткой формуле:

**Alaya не просто помогает агенту помнить.  
Она помогает агенту вести состояние мира и вовремя действовать.**

Или еще короче:

**From conversations to state. From state to action.**

## Заключение

Alaya имеет смысл мыслить не как "бота" и не как "очередную AI memory database".

Более сильная рамка звучит так:

**Alaya — это слой операционной памяти и workflow-исполнения для AI-ассистентов.**

Его задача:

- принимать события
- собирать структурированное состояние
- различать факт и гипотезу
- хранить provenance
- отслеживать изменения во времени
- запускать нужные workflows

Если эта рамка будет доведена до ясного core, она может одинаково хорошо лечь и под корпоративного ассистента, и под relationship copilot, и под другие memory-native продукты.
