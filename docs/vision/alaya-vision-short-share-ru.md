# Alaya / Between — короткая версия vision для пересылки

Черновик короткой версии  
Язык: русский  
Назначение: быстро объяснить идею людям без полного технического контекста

## Что это вообще такое

Главная идея Alaya и связанных с ней продуктов не в том, чтобы просто дать агенту “память”.

Более точная формулировка такая:

**Alaya — это операционная память и движок workflows для AI-ассистентов.**

Если совсем просто:

**система принимает поток разговоров и событий, превращает их в рабочее состояние мира и помогает действовать на основе этого состояния.**

Короткая формула:

**Из разговоров в состояние. Из состояния в действия.**

## Почему этого недостаточно назвать просто memory

Во многих AI-системах “память” означает:

- история сообщений
- векторный поиск по прошлому
- набор сохраненных фактов

Но для реальных рабочих и личных сценариев этого мало.

Нужно уметь:

- понимать, что было обещано
- видеть, что тема осталась незакрытой
- отслеживать статусы во времени
- различать факт и предположение модели
- вовремя напоминать
- запускать follow-up действия

То есть нужна не просто память, а **рабочая модель мира + execution layer**.

## Как это устроено в простой схеме

### 1. Raw events

Система получает сырые события:

- чаты
- звонки
- транскрипты
- документы
- обновления из интеграций

### 2. Extraction

Из сырых событий извлекаются:

- сущности
- факты
- связи
- обязательства
- решения
- риски
- статусы

### 3. Canonical state

Полученные данные собираются в рабочую модель мира:

- кто есть кто
- какие есть объекты
- что подтверждено
- что только inferred
- что устарело
- что требует follow-up

### 4. Workflow layer

Когда состояние меняется, могут запускаться действия:

- reminder
- review item
- recurring workflow
- delayed task
- follow-up

Именно этот слой делает систему operational.

## Где это особенно полезно

Такая архитектура особенно полезна там, где важны:

- обязательства
- незакрытые темы
- изменения состояния во времени
- follow-through
- reminders
- много разных источников данных

Поэтому один и тот же core потенциально может лежать под разными продуктами:

- корпоративный AI chief of staff
- relationship copilot
- personal assistant
- coaching assistant
- memory-native workflow tools

## Как это связано с AlayaOS и Between

Сейчас разумно смотреть на это так:

- `AlayaOS` — текущий B2B-продукт
- `Between` — новый personal / relationship vertical
- `Alaya Core` — возможное будущее общее ядро, если оно реально окажется полезным обоим продуктам

Важно:

**shared core пока считается гипотезой, а не отдельным продуктом.**

Его не надо строить в отрыве от реальных use cases.

## Главный вывод

Сильная версия этой идеи звучит так:

**Мы строим не просто память для LLM.  
Мы строим систему, которая из потока событий собирает модель мира и умеет действовать по изменениям этой модели.**

Именно это отличает направление Alaya от обычного RAG, от chat history memory и от многих generic memory layers.
